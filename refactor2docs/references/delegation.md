# Refactor2Docs - 并行委派协议

## 何时委派
- 模块数 > 5，且模块目录边界清晰。
- 可以按目录/子系统分割，互不写同一文档文件。
- 主 Agent 已有全局模块清单和命名规则。

## 不应委派
- 入口/架构边界尚不清楚。
- 多个子任务会分析同一模块并产生冲突。
- 需要统一判断的架构决策、文档目录设计、最终验证。

## 子 Agent 任务模板
```text
分析 {module_name} 模块，范围仅限 {path}。

输出：
1. 模块职责与边界。
2. 核心类/函数/接口/数据模型，必须带源码路径，关键结论到行号。
3. 上游调用方、下游依赖、外部 IO。
4. 关键执行流程和异常路径。
5. 现有测试与目标测试建议。
6. 风险、重构注意事项、未确认问题。

禁止：跨目录扩展、编造业务背景、修改文件。
完成标准：发现的关键符号均被分类；零结果明确写“未发现”。
```

## 合并协议
- 去重：相同结论保留证据更具体的一条。
- 冲突：标记为“需复核”，附双方证据。
- 计数：子模块数量、符号数量、依赖边数量要能汇总到全局表。
- 边界：跨模块依赖由主 Agent 统一命名和归类。
- 验证：最终由主 Agent 对所有文档执行 `references/verification.md`。
