# Refactor2Docs - 领域知识与八层文档架构

## 目标
为旧代码重构建立“可理解、可验证、可迁移”的文档基线。文档不是总结，而是重构资产：后续实现者应能凭文档理解边界、复写接口、设计测试并评估变更影响。

## 每层最低内容标准

| 层 | 必须回答 | 证据要求 |
|---|---|---|
| 01 需求文档 | 系统/模块对外提供什么能力、用户/调用方是谁、验收条件是什么 | 从路由、CLI、API、UI、测试、配置反推；不确定业务意图要标注 |
| 02 架构设计 | 系统如何分层、模块如何协作、关键设计决策是什么 | 入口、依赖、配置、主要调用链 |
| 03 模块设计 | 模块内部类/函数/状态/数据模型/异常如何工作 | 关键源码路径，重要结论到行号 |
| 04 测试用例 | 如何锁定旧行为并支持重构 | 现有测试 + 从代码行为提取的新测试场景 |
| 05 构建部署 | 如何安装、构建、运行、部署、回滚 | package/lockfile/CI/Docker/config/scripts |
| 06 重构指南 | 如何分批重构、先改哪里、风险和回滚是什么 | 耦合点、依赖图、测试缺口、变更影响 |
| 07 技能文档 | 本次分析方法、约束、复用经验 | 本次产物与命令/工具证据 |
| 08 代码图谱 | 模块、符号、流程和数据如何连接 | 静态依赖、调用关系、入口到核心流程 |

## 证据标注规范
- 确认事实：`path/to/file.ext:line` 或 `path/to/file.ext`。
- 推断事实：`基于 path/to/file.ext:line 的命名/流程推测`。
- 未发现事实：写 `未发现`，并说明搜索范围或依据。
- 文档内部要优先引用相对路径，便于迁移。

## 分析顺序建议
1. 先找入口和配置：README、package/build、CI、main/index、routes、commands、controllers。
2. 再找稳定边界：模块目录、公开 API、适配器、服务层、数据层。
3. 然后追关键流：用户入口 → 协调层 → 业务逻辑 → 外部 IO/状态变化。
4. 最后写重构资产：测试规格、影响分析、分批计划、代码图谱。
