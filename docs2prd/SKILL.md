# Docs → PRD 技能

将技术文章/文档转化为符合 IEEE 830 规范的产品需求规格说明书（PRD）。

**输入：** 任意技术文章（微信文章/公众号/博客/论文/技术文档）的 URL 或原始文本
**输出：** 结构完整的 PRD 文档（`.md` 格式），可直接存入 `~/wiki/queries/`

**固化背景：** 本技能源于从微信公众号《万字干货！Harness Engineering如何工程化落地？》一文转化为 PRD 的真实操作经验。微信防爬机制处理、8维知识抽取框架、IEEE 830结构化建模方法，均在此过程中验证过。

---

## 何时激活本技能

用户要求将某篇文章/文档转化为 PRD 时激活，包括但不限于：
- "把 XXX 文章转成 PRD"
- "将 XXX 技术文档写成产品需求文档"
- "帮我写一个 XXX 的规格说明书"
- "这篇文章的产品需求是什么"

### 强制前置规范（铁则，违反则立即停止）

### 信息溯源铁则
- **所有核心内容必须100%来自原文**，不得捏造、推测业务逻辑
- **补充内容**（如通用工程规范、行业标准要求）必须标注：`【原文未明确提及，按软件工程行业通用规范补充】`
- 每个核心论点必须能找到对应的原文章节/段落
- **溯源粒度要求**：`【原文第X章/X.X节】`（禁止只用行号）；如原文无章节结构，方可用行号替代但需注明"原文行号"

### 流程不可跳步铁则
必须严格按四步执行：**解析预处理 → 8维抽取 → 结构化建模 → PRD生成**
禁止合并步骤、禁止简化步骤、禁止在未完成上一步之前进行下一步

### 规范强制遵循铁则
- 输出**必须**遵循 IEEE 830 软件需求规范
- 架构描述**必须**采用 C4 模型（C4Context + C4Container），**禁止使用ASCII art方框图替代Mermaid C4图**
- 8大章+子节结构**禁止删减**
- **每章必须有实质内容**，禁止生成"【内容由通用规范补充】"的空壳章节

### 验收标准量化铁则
- 每条验收标准必须可验证、可量化
- 禁止模糊表述（"尽量"/"较好"/"提升"）
- 使用具体数值或比率

### PRD文档头部模板（必须严格遵循）
每个PRD文档开头**必须**包含以下格式的元数据块（直接写在文件顶部，不使用YAML frontmatter）：

```
# 产品需求规格说明书（PRD）
**文档名称：** {产品名称}
**版本：** V1.0
**日期：** {YYYY-MM-DD}
**原文来源：** {原文标题}（{发布平台}，{作者}）
**原文链接：** {URL}
**信息溯源说明：** 本文档所有核心内容100%来自上述原文。信息溯源详见附录8.1原文核心信息抽取对照表。原文未明确提及的内容，按软件工程行业通用规范补充，并标注【原文未明确提及，按软件工程行业通用规范补充】。

---
```

## 四步执行流程

### Step 0：获取原文内容

#### 微信公众平台文章

微信文章有防爬验证（`js_pc_qr_code` 节点），但正文内容实际已完整存储在 HTML 的 `id="js_content"` 区块中，可直接提取：

```python
import urllib.request
import re
import html

url = "https://mp.weixin.qq.com/s/..."

req = urllib.request.Request(url, headers={
    "User-Agent": "Mozilla/5.0 (Windows NT 10.0; Win64; x64) AppleWebKit/537.36",
    "Accept": "text/html,application/xhtml+xml",
})
with urllib.request.urlopen(req, timeout=30) as r:
    html_content = r.read().decode("utf-8", errors="replace")

# 提取 js_content 区块
match = re.search(r'id="js_content"[^>]*>(.*?)</section>', html_content, re.DOTALL)
if match:
    raw_html = match.group(1)
    # 清理 HTML 标签，保留段落换行
    text = re.sub(r'<[^>]+>', '', raw_html)
    text = html.unescape(text)
    # 合并多余空行
    text = re.sub(r'\n{3,}', '\n\n', text)
    print(text[:500])
```

**验证是否成功抓取：**
- 文件大小正常（微信文章通常 100KB~5MB HTML）
- 包含正文内容（章节标题、段落）
- 不含"环境异常"/"验证异常"等验证拦截提示

#### 普通网页文章

优先使用 `web_extract`（MCP工具）直接获取 markdown：
```
web_extract 工具获取 markdown 正文
```
若 `web_extract` 失败或返回质量不佳，再用 curl + BeautifulSoup 方案。

#### PDF/论文

使用 `web_extract`（支持 PDF 解析），或：
```bash
curl -L "URL" -o article.pdf
# 使用 pdf_ocr_documents skill 提取文本
```

#### 用户直接粘贴文本

直接使用粘贴内容，无需抓取。

### Step 1：解析预处理

**目标：** 将原始文本去噪、识别文章结构、定位核心逻辑线

**操作步骤：**

1. **全文通读**（分 500 行/次读取，确保不遗漏）
2. **识别文章类型**（决定溯源标记格式）：
   - **论文体**：有编号章节（1.1/2.3）→ 溯源格式 `【原文第X章/X.X节】`
   - **博客体**：有markdown标题但无编号 → 溯源格式 `【原文"{标题}"】`
   - **访谈/案例体**：对话/访谈为主 → 溯源格式 `【原文"{话题}"段落】`
   - **文档体**：原文本身就是需求/规格文档 → 溯源格式 `【原文第X条/{条款名}】`
3. **识别章节结构**：数清章节数量，记录每个章节的核心主题
4. **过滤无效信息**：
   - 文末广告（腾讯云推广、扫码领券、赠品活动等）
   - 作者声明、关注引导
   - 平台水印
5. **定位核心逻辑线**：文章解决什么业务问题 / 提出了什么方案 / 架构是什么

**输出：** 提纯文本（`/tmp/article_purified.txt`），行数控制在合理范围（通常 500~2000 行）

**质量检验：** 提纯文本应只包含正文内容，不含任何 HTML、广告、噪声字符。

---

### Step 2：8维知识精准抽取

**目标：** 从原文中抽取8个维度的信息，为结构化建模做准备

**8个维度定义：**

| 维度 | 抽取内容 | 常见遗漏点 |
|------|----------|-----------|
| **业务背景层** | 痛点、市场/业务/技术问题、目标用户、使用场景 | 作者的背景设定（行业/规模/团队特征）容易被忽略 |
| **需求层** | 功能性需求（FR）、非功能性需求（NFR）、数据需求 | 原文无"需求"字眼但实际构成约束的内容（"应该"/"必须"等描述性语句）容易被遗漏 |
| **方案层** | 核心方案原理、技术选型对比、核心创新点 | 方案优势对比（原文明确说"选A不选B"的原因）容易被缩减为结论 |
| **架构层** | 系统架构图、模块拆分、模块间交互关系 | 模块间关系和交互协议容易被省略 |
| **数据层** | 数据来源、数据格式、数据存储要求 | 数据安全/合规要求容易被留空 |
| **接口层** | 内部接口（模块间）、外部接口（第三方系统） | 外部依赖系统的接入方式容易被忽略 |
| **边界层** | 系统边界、约束条件、不适用于哪些场景 | "不适用于"的排除场景容易被遗漏 |
| **术语层** | 专有名词定义、缩写表 | **原文明确解释的术语必须收录**，不得用个人理解替代；缩写表必须100%来自原文定义 |

**操作步骤：**

1. **逐维度分析**：对每个维度，从提纯文本中找到对应内容
2. **抽取核心内容**：不要总结，用原文表述 + 对应章节溯源标记
3. **识别功能性需求**：识别动词短语（"系统应支持..."、"系统必须能够..."）——这些是FR的候选
4. **识别非功能性需求**：识别性能、精度、安全性、可用性相关描述
5. **记录溯源**：每条核心内容标注 `【原文第X章/X.X节】`（必须精确到节，不能只标行号）
6. **术语100%覆盖**：原文中有定义/解释的术语必须全部收录到术语表，不得遗漏，不得自行定义或替换原文术语

**输出：** 8维抽取对照表（结构化文本，便于 Step 3 直接引用）

**常见陷阱：**
- 把作者的个人观点当作客观需求 → 区分"原文明确说"和"原文暗示"
- 遗漏隐性需求（如"上下文纪律"这种未用"需求"字眼但实际构成约束的内容）
- 过度引申：原文中AI/软件工程领域通用知识不需要溯源标注，但需标注"按通用规范补充"

### Step 3：结构化工程建模

**目标：** 将8维抽取结果按 IEEE 830 + C4 模型进行结构化建模

#### 3.1 需求建模（IEEE 830）

按 IEEE 830-1998 标准结构化：

```
1. 文档概述
   1.1 目的与适用范围
   1.2 原文来源与信息溯源说明（含溯源对照表）
   1.3 术语定义与缩写表
   1.4 参考文档

2. 业务背景与产品定位
   2.1 市场/业务/技术痛点（100%来自原文）
   2.2 核心业务目标与产品定位
   2.3 目标用户与核心使用场景
   2.4 约束条件

3. 需求规范（IEEE 830核心）
   3.1 整体需求概述
   3.2 功能性需求（FR-001, FR-002...）
       每条含：需求ID/描述/前置条件/后置条件/用户场景/验收标准
   3.3 非功能性需求（NFR-001...）
       含量化指标：MTBF/遵守率/准确率等
   3.4 数据需求（来源/格式/存储/安全）
   3.5 接口需求（内外部接口）
   3.6 其他需求

4. 总体方案设计与架构规范（C4）
   4.1 核心方案原理与技术选型说明
   4.2 系统上下文架构（C4 Level1）
   4.3 系统容器架构（C4 Level2）
   4.4 核心技术流程与端到端数据流设计
   4.5 核心创新点与方案优势

5. 模块框架与详细设计
   5.1 核心模块整体拆分说明
   5.2 各模块详细设计（每模块含：定位/职责/输入/输出/核心功能/依赖关系）
   5.3 模块间交互逻辑与时序设计（含 Mermaid 时序图）

6. 验收标准与测试要求
   6.1 功能验收标准（表格，含测试方法）
   6.2 性能与效果验收标准（量化指标）
   6.3 测试范围与测试类型要求

7. 风险评估与规避方案
   每条含：风险ID/描述/发生概率/影响程度/规避方案

8. 附录
   8.1 原文核心信息抽取对照表
   8.2 架构图、流程图可视化说明（Mermaid代码）
```

#### 3.2 架构建模（C4）

**警告：必须使用Mermaid C4语法，禁止使用ASCII art方框图。**

微信等平台文章通常无配图，C4图需根据原文描述**推断生成**（推断内容需标注`【按C4建模规范补充】`）。Mermaid C4语法：

```
```mermaid
C4Context
    title 系统上下文架构 - {产品名称}
    Person(usr, "用户", "角色描述")
    System(sys, "{系统名}", "系统职责")
    SystemDb(db, "{数据库}", "存储内容")
    Rel(usr, sys, "交互关系", "协议")
    Rel(sys, db, "读写", "协议")
C4Container
    title 容器架构 - {产品名称}
    Container(ctrl, "{组件名}", "技术栈", "核心职责")
    Container(svc, "{服务名}", "技术栈", "核心职责")
    Rel(ctrl, svc, "调用/消息", "协议")
```

**Level1（上下文图）描述规则：**
- 必须包含：`Person`（人）、`System`（系统）、`SystemDb`（数据库）三类元素
- 必须标注系统边界 `System_Boundary`
- 每条关系用 `Rel` 标注，说明 interaction + protocol

**Level2（容器图）描述规则：**
- 每个容器是一个可独立部署/运行的单元
- 必须用 `Container_Boundary` 划分层次（如"交互渠道层"/"核心服务层"/"数据存储层"）
- 必须标注技术栈（用引号）

**常见错误：**
- ❌ 用ASCII方框+Unicode线段绘制架构图（无法渲染，且不符合C4规范）
- ❌ 只有System没有Person（缺失系统用户视角）
- ❌ 关系无label（缺少interaction描述）
- ❌ 容器图缺少技术栈标注

#### 3.3 流程建模

**主流程：** Mermaid flowchart（TD 模式），展示阶段接力、打回路径、交付条件。

**模块时序图：** Mermaid sequenceDiagram（可选，用于复杂模块间交互）。

#### 3.4 边界建模

**系统上下文：** 明确什么是系统内的，什么是外部依赖（扩展阶段）。

### Step 4：PRD 生成与自检

**生成：** 将 Step 3 的建模结果写入 `~/wiki/queries/prd-{slug}-{date}.md`

**强制自检清单（10条，每条必须验证）：**

| # | 检查项 | 通过标准 |
|---|--------|----------|
| 1 | 信息溯源 | 核心内容100%来自原文；补充内容标注【按通用规范补充】；溯源精确到章节/小节而非仅行号 |
| 2 | 流程完整性 | 四步全部执行，无跳步 |
| 3 | IEEE 830 结构 | 8大章全部存在，无缺章 |
| 4 | C4 架构图 | **必须使用Mermaid C4语法（C4Context + C4Container）**；禁止使用ASCII art方框图替代 |
| 5 | 量化验收标准 | 每条标准可量化，无模糊词 |
| 6 | 术语表 | 原文明确解释的术语100%收录，无遗漏，无自行定义或替换 |
| 7 | 原文链接 | 文档头部元数据块包含完整原文URL |
| 8 | Mermaid 可渲染 | 所有图表为有效 Mermaid 代码 |
| 9 | 无空壳章节 | 每章必须有实质内容，不得生成"【内容由通用规范补充】"的空壳 |
| 10 | 文档头部格式 | 必须使用标准头部模板（文档名称/版本/日期/原文来源/原文链接/溯源说明） |

**如有任何一条未通过，立即修复后再输出。**

## PRD 文件名命名规范（强制）

**格式：**
```
{产品名}-{产品类型}-{文档类型}-{YYYYMMDD}.prd
```

**各部分提取规则：**

| 部分 | 规则 | 示例 |
|------|------|------|
| **产品名** | 原文明确提及的产品/项目名称，去除任何前缀修饰 | Onyx / Claude Code / Meta-Harness |
| **产品类型** | 根据产品本质归纳（3~6字）：知识库、工程平台、测试系统、编程工具、CI系统等 | 开源AI知识库 / 工程平台 / 测试Agent系统 |
| **文档类型** | 本次PRD的文档性质：需求方案分析 / 产品需求规格说明书 / 技术规格说明书 | 需求方案分析 / 产品需求规格说明书 |
| **日期** | 转化日期，YYYYMMDD格式（与PRD文件内日期一致） | 20260429 |

**禁止出现的内容：**
- ❌ 媒体化修饰词：怒锤、震撼、揭秘、终极、干货、史诗
- ❌ 平台标签：微信公众号、知乎、原创等
- ❌ 作者名（除非产品名本身就包含作者）
- ❌ 感叹号、星号等符号
- ❌ 原文标题照搬（原文标题是给读者看的，PRD文件名是给工程师检索的）

**Good vs Bad 示例：**

| 原文标题 | ❌ 错误PRD命名 | ✅ 正确PRD命名 |
|----------|--------------|----------------|
| 开发者怒锤Anthropic，Onyx吊打Claude | Onyx-开发者怒锤Anthropic-20260429.prd | Onyx-开源AI知识库-需求方案分析-20260429.prd |
| 万字干货！Harness Engineering如何落地？ | prd-万字干货！Harness-20260429.md | Harness-工程平台-产品需求规格说明书-20260429.prd |
| 整个团队0人工Coding实战手册 | prd-0人工Coding实战手册.md | 团队编程平台-需求方案分析-20260429.prd |
| 构建会思考的测试Agent | prd-会思考的测试Agent-20260429.md | 测试Agent系统-产品需求规格说明书-20260429.prd |

**对应修改：**
- PRD文件保存时，文件名使用上述规范（`.prd`后缀）
- Wiki存档时，文件名同步使用此规范
- Wiki index.md 中的标题同步更新

## Wiki 存档流程

PRD 生成完成后，必须同步更新 Wiki：

1. **归档原文**：将提纯文本保存到 `~/wiki/raw/articles/{产品名}-{日期}.txt`
2. **写入 PRD**：将 PRD 保存到 `~/wiki/queries/{产品名}-{产品类型}-{文档类型}-{YYYYMMDD}.prd`
3. **更新 index.md**：在 Queries 章节添加条目
4. **追加 log.md**：
```
## [YYYY-MM-DD] ingest | {产品名} {文档类型}
- Source: {原文URL}
- Output: ~/wiki/queries/{产品名}-{产品类型}-{文档类型}-{YYYYMMDD}.prd
```

## 关键陷阱与应对

### 陷阱1：微信防爬验证误判为抓取失败
- **现象**：`js_pc_qr_code` 节点出现，但正文内容实际完整
- **判断**：检查 HTML 文件大小（正常应 500KB~5MB），检查是否含正文章节内容
- **应对**：直接正则提取 `id="js_content"` 区块即可，无需执行 JS

### 陷阱2：Rule 与 Scripts 混淆
- **Rule**：软约束，AI 尝试遵守但会绕过，是"原则性约束"
- **Scripts**：硬门禁，"脚本判定通过才算完"，是"执行性约束"
- 两者必须严格区分，不能互相替代

### 陷阱3：角色越界难以识别
- PM 只做路由（越权提供专业意见 = 0容忍）
- 下游不能直接改上游文档（越界修改 = 0容忍）
- 这些是流程红线，必须在架构层面控制，而非依赖 AI 自觉

### 陷阱4：上下文纪律被忽视
- 每棒交接只给当前该看的那一份材料（不是瞒信息，是避免上下文过长）
- dev-map 和任务看板在流程中同步更新，而非事后一次性整理

### 陷阱5：技术选型论据不充分
- 任何技术选型结论必须有原文对应支撑（原文核心论据 or 原文数据）
- 不能用"通常"/"业界普遍认为"替代原文溯源

### 陷阱6：NFR 指标全靠补充
- **正确做法**：原文有量化指标的用原文；原文无量化指标的才用【按通用规范补充】
- **常见错误**：所有 NFR 都标注"按通用规范补充"，忽略原文实际提到的指标

### 陷阱7：文章无章节结构时溯源粒度失控
- **现象**：微信文章/博客常无1.1/2.3等层级章节，仅有自然段落或markdown标题
- **应对**：先识别文章的结构特征（是论文体/博客体/访谈体/文档体），再决定溯源标记格式
  - **论文体**（有编号章节）：溯源格式 `【原文第X章/X.X节】`
  - **博客体**（有markdown标题但无编号）：溯源格式 `【原文"{标题}"】`
  - **访谈/案例体**（对话为主）：溯源格式 `【原文"{话题}"段落】`
  - **文档体**（原文本身是需求文档）：溯源格式 `【原文第X条/{条款名}】`

### 陷阱8：PRD章节内容空洞化
- **现象**：章节内容全部是"【按通用规范补充】"而无原文实质内容
- **原因**：原文本身未覆盖该维度（如竞品分析文章通常无架构设计章节）
- **应对**：
  - 若是原文覆盖不足，该章节应标注"【原文未覆盖此维度，详见补充说明】"，保留框架但明确来源
  - 若是原文完全未提及（如论文原文），则该章节用通用规范补充，但必须说明"本文来自{文章类型}，{维度}为按IEEE 830通用规范建模"

### 陷阱9：术语表遗漏原文术语
- **现象**：原文有明确定义的技术术语（如"MCP"/"Rule"/"Skill"），在PRD中未被收入术语表
- **原因**：仅摘取了"核心概念"而遗漏了术语表维度的系统性抽取
- **应对**：Step 2的术语层抽取必须作为独立步骤执行，不能合并到方案层

## 快速参考

**PRD 固定结构（8大章）：**
```
一、文档概述
二、业务背景与产品定位
三、需求规范（IEEE 830核心）
四、总体方案设计与架构规范（C4）
五、模块框架与详细设计
六、验收标准与测试要求
七、风险评估与规避方案
八、附录
```

**8个需求维度：** 业务背景层 / 需求层 / 方案层 / 架构层 / 数据层 / 接口层 / 边界层 / 术语层

**C4 四级：** Context（上下文）→ Container（容器）→ Component（组件）→ Code（代码）

**IEEE 830 核心要素：** 功能性需求（FR-ID格式）+ 非功能性需求（NFR-ID格式）+ 每条含验收标准

**溯源标记格式：** `【原文第X章/X.X节】` 或 `【原文未明确提及，按软件工程行业通用规范补充】`
